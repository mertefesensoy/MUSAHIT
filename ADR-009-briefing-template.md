## ❯ Context

The briefing is the operator's daily artifact · the single thing read every morning with
coffee. Its structure shapes the operator's mental model of what happened in Turkey. The
template must be optimized for skim-and-stop: the operator reads top-down, absorbs the
highest-priority items first, and can stop at any depth with the most important content
already in mind.

The briefing is also the input to the TTS stage (ADR-010), which voices DEFCON 1-3 items
plus open arc updates. The template must be parseable enough that the TTS stage can
extract the voiced portion without re-running the writer.

## ❯ Decision

The briefing is generated by the writer model (Trendyol-LLM 7B) per a structured Turkish
template. The output is a Markdown document with explicit section markers that downstream
stages parse. An HTML version is rendered from the Markdown for the dashboard.

### Writer prompt structure

The writer prompt is constructed by `src/writer/prompt.py`. It contains:

1. **System role** · "Sen MÜŞAHİT'in yazar modelisin. Türkiye gündemini tarafsız ve
   resmi Türkçeyle özetlersin. Görüş bildirmezsin. Olguları kaynak atıflarıyla
   sunarsın. Farklı bantlar farklı çerçeveler kullandığında ikisini de gösterirsin."
2. **DEFCON schema reminder** · the schema and Turkish labels from ADR-004
3. **The day's clusters** · structured JSON · grouped by final DEFCON and category ·
   each cluster includes headline, summary, member articles with band tags, entities
4. **Open arc updates** · arcs with new clusters today · current summary + new
   development
5. **Resolved arcs** · arcs transitioning to RESOLVED today
6. **System log data** · for the footer
7. **Output instruction** · "Yukarıdaki şablonu doldur. Her bölümü Türkçe yaz. Şablonun
   yapısını değiştirme."

### Discipline rules (in prompt)

- **No editorializing** · the writer reports · does not opine
- **Cross-band framing** · when gov-aligned and opposition outlets frame the same event
  differently, both framings appear with `KAYNAK·BAND` attribution
- **Quote sparingly** · at most one direct quote per item · under 15 words · in
  quotation marks · with attribution
- **Confidence is shown** · every item carries `Güven · YÜKSEK/ORTA/DÜŞÜK`
- **Arc linkage is shown** · every item that belongs to an arc references it by id
- **Formal Turkish** · "resmi Türkçe" register · no slang · no English loanwords where
  Turkish alternatives exist

### Skim-and-stop optimization

The template is ordered so that the operator can stop reading at any section break and
have absorbed the most important content. This means:

- DEFCON 1-2 always come first
- Open arcs come before routine items because continuity matters more than novelty
- DEFCON 4 is compact (one-line bullets) because the operator has by this point absorbed
  the day's priority items
- AMBİYANS is intentionally NOT listed in the briefing body (only counted) because if
  the operator gets here, they're done

### TTS scope

Per ADR-010, the TTS engine reads only:

- The header (Tarih, Zirve DEFCON, İşlenen olay)
- DEFCON 1-2 · ÖNCELİKLİ (full items including direct quotes)
- DEFCON 3 · MATERYAL (one-paragraph summaries, no source lists)
- AÇIK GELİŞMELER · DEVAM EDEN TAKİP · **only the `### Öne Çıkanlar`
  subsection · capped at `VOICED_OPEN_ARCS_CAP = 10` arcs** (2026-05-24)
- A closing line "DEFCON 4 ve sonrası dashboard'da görüntülenebilir."

DEFCON 4, DİKKAT, AMBİYANS, KAPATILAN HİKAYELER, and SİSTEM LOG are NOT voiced. The
operator reads them on the dashboard.

Estimated audio length · 3-4 minutes (post-2026-05-24 · was 4-6).

#### AÇIK GELİŞMELER subsection split (2026-05-24)

The section now has two subsections rendered by both the fallback
renderer and (when retried) the writer LLM:

* `### Öne Çıkanlar` · the top `VOICED_OPEN_ARCS_CAP = 10` arcs by
  `(peak_defcon ASC, last_update_at DESC)`. Lower peak_defcon = more
  severe per ADR-005 / ADR-008. Tie-broken by most-recent update.
  Each arc rendered with full `### headline · arc_id` block, peak
  DEFCON, category, and (when present) summary. **This subsection is
  what the TTS extractor voices.**
* `### Diğer Açık Hikayeler` · all remaining open arcs as a one-line
  bullet list `- {headline} · {DEFCON_LABEL_TR} · {category} ·
  \`{arc_id}\``. **Visual-only · the TTS extractor stops bucketing
  voiced content once it encounters this marker.**

When the total open arc count is `≤ VOICED_OPEN_ARCS_CAP`, only the
Öne Çıkanlar subsection is rendered (no Diğer subsection at all).
This keeps the briefing visually clean on quiet days.

Rationale: the briefing is consumed at 07:00 with coffee · a 3-4
minute audio is the long edge of what an operator listens to before
switching context. The smoke run on 2026-05-23 produced a 64K-char
chunk that Piper timed out on (60s per-chunk budget per ADR-010);
without the cap, every run scales with the open arc backlog and
eventually breaks TTS. The cap of 10 is the smallest number that
still surfaces the operator's full priority queue on heavy days
(the priority queue is itself bounded by the arc-link stage's
severity ordering).

### Dashboard rendering

The HTML version of the briefing applies the PoI aesthetic per ADR-011:

- Monospace font (IBM Plex Mono or JetBrains Mono)
- Dark background (`#0a0a0a`) with amber text for DEFCON 1-2 (`#ffb000`) and white text
  for everything else
- Section dividers as ASCII rules (`──`)
- Inline severity badges
- Click any source attribution to expand the cluster's article list
- Click any arc id to open the arc detail view

### Operator interactions on the briefing

The HTML briefing is interactive. The operator can:

- Click `PROMOTE`/`DEMOTE` next to any item to override its DEFCON
- Click `RESOLVE` next to any arc to close it
- Click `MERGE` between two items to consolidate them
- Click `MARK READ` at the bottom to dismiss the briefing

All actions write to `manual_overrides`.

## ❯ Consequences

**Positive**
- The template is consistent night-to-night · operator builds skim muscle memory
- Bias discipline is in the writer prompt · the model is constrained, not the output
- TTS scope is explicit · no ambiguity about what gets voiced
- HTML version is the operator's primary interface · markdown is the parseable source
  of truth

**Negative**
- Writer model must hold the entire template structure · prompt is ~3-4K tokens · with
  the day's clusters this can push to 12-16K input tokens · acceptable for Trendyol-LLM
- If writer model drifts from template (skips sections, reorders), parsing breaks ·
  mitigation: a template validator runs after generation · if invalid, the writer is
  re-prompted with the specific error

## ❯ Alternatives considered

- **Inline template (single Markdown blob without section markers)** · rejected · TTS
  stage needs structure
- **Separate prompts for each section** · over-engineered · single prompt is more
  coherent
- **English-language briefing** · rejected · operator is a Turkish speaker · TTS is
  Turkish · output must be Turkish

## ❯ Open questions

- Writer model context window may need experimentation · if Trendyol-LLM 7B can't hold
  a full day's worth of clusters, the prompt may need to be sectioned
- Whether to add a `KARŞIT GÖRÜŞ` (counterpoint) section that explicitly highlights
  where bands disagree most sharply · deferred
