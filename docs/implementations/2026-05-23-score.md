# Implementation: Score stage · `musahit/score/`

**Date** · 2026-05-23
**Author** · Claude Code (Mert Efe Şensoy directing)
**ADR refs** · ADR-002 · ADR-004 · ADR-005 · ADR-006 · ADR-012

---

## ❯ Problem / Motivation

Build step 11 of 20. Step 10 produced clusters (groups of articles
reporting the same event); this stage assigns each cluster a severity
score (DEFCON 0-5 per ADR-004) and a confidence tag, applies the
bias-promotion ceiling rules from ADR-005, and writes the audit row to
`promotion_log`. The cluster row's `final_defcon` then drives every
downstream decision: briefing template order, audio inclusion (DEFCON
1-3 only), dashboard tab placement, peak-DEFCON arc bookkeeping.

Three things differentiate this from previous stages:

1. **First worker-LLM stage.** Qwen2.5 7B via Ollama produces the
   `raw_defcon`. The Ollama integration mirrors the embedder Protocol
   established in step 10: an `LlmClient` Protocol with an
   `OllamaLlmClient` production impl and a `FakeLlmClient` for tests.
2. **Two-step architecture.** The LLM is non-deterministic; the
   promotion rules are deterministic. The audit trail (promotion_log)
   needs both. We separate `WorkerResponse` (LLM output) from the
   promotion arithmetic so each is independently testable.
3. **First FILE-PROTECTED `score/` files.** `defcon.py` and
   `promotion.py` enter the protected list. Both encode contracts that
   downstream stages depend on (briefing template slots, audio cutoffs,
   promotion-rule auditability); both are explicitly locked.

## ❯ Implementation Approach

### Two-step architecture

```text
        ┌──────────────────────┐
        │  Step 1: Worker LLM  │  (non-deterministic)
        │                      │
        │  prompt + cluster ──▶│   raw JSON
        │                      │
        │   WorkerResponse     │
        │   (pydantic)         │
        └─────────┬────────────┘
                  │ raw_defcon
                  ▼
        ┌──────────────────────┐
        │ Step 2: Promotion    │  (deterministic)
        │                      │
        │  bands_present  ────▶│   ceiling_defcon
        │  num_sources         │   sides_present
        │                      │   confidence
        │                      │
        │  final = min(raw,    │   rule_applied
        │              ceiling)│
        └─────────┬────────────┘
                  ▼
        clusters row + promotion_log row
```

The worker decides "how severe is this event?". The promotion rules
decide "how much can the cluster's band coverage support that
severity?". Both decisions are stored. The operator can audit either
half independently.

### Prompt builder reading anchors

`build_classification_prompt(articles)` reads `DEFCON_ANCHORS` and
`DEFCON_LABEL_TR` from `defcon.py` directly. Single source of truth: an
ADR-004 amendment that updates an anchor automatically flows into every
classification call, no copy-paste of anchor strings.

The prompt is deterministic for a given set of articles — no clocks,
no random salt, no per-run nonce. That keeps `FakeLlmClient`'s
substring-keyed response map usable for tests.

### Pydantic + retry pattern

```python
async def _classify_one(cluster) -> tuple[WorkerResponse, bool]:
    prompt = build_classification_prompt(cluster.articles)
    attempt = 0
    while attempt <= self._max_retries:
        raw = await self._llm.generate(prompt)
        try:
            return parse_worker_response(raw), False
        except ValidationError:
            attempt += 1
    return _FALLBACK_RESPONSE, True
```

`parse_worker_response` strips markdown code fences before
JSON-decoding (Qwen sometimes wraps its output in ` ```json ` blocks
even when instructed not to). JSON-decode failures are re-raised as
`ValidationError` so the retry loop has one exception type to catch.

The fallback is `{defcon: 5 (AMBIENT), category: UNCLASSIFIED,
confidence_self: low}`. It mirrors the ADR-012 "degrade gracefully"
principle: the briefing shows the cluster, demoted, with the
SİSTEM LOG footer surfacing the score gap.

### Bootstrap demotion via pipeline_runs lookup

`bootstrap_demoted(cluster.created_at, conn)` queries
`MIN(pipeline_runs.started_at)` to find the system's first run. If the
cluster falls within 7 days of that, `apply_bootstrap_demotion(final)`
bumps `final` by one tier toward AMBIENT (capped at AMBIENT — can't
demote further).

The 7-day window is ADR-004's calibration buffer. During the first
week the worker model's calibration is unproven; the operator's manual
overrides during this period inform tuning. The bootstrap demotion
protects against over-promoting events while we learn the operator's
preferences.

Lookup-via-pipeline_runs avoids a config flag: the bootstrap window is
defined by data, not by manual operator state.

### FakeLlmClient template

The fake mirrors the embedder's fake from step 10. Two modes:

```python
# Substring responses — happy-path test.
llm = FakeLlmClient(default='{"defcon": 3, ...}')

# Callable responder — retry/fail-path test.
def responder(prompt, attempt):
    if attempt == 0:
        return "not json"
    return '{"defcon": 4, ...}'
llm = FakeLlmClient(responder=responder)
```

Use this template for every future Ollama-using stage (arc-link
worker, writer). The `(prompt, attempt)` signature is what makes
retry-on-fail tests trivial.

### DuckDB foreign-key workaround

DuckDB 1.4.2 fires a foreign-key constraint error on `UPDATE clusters
SET ...` when `cluster_articles` (or `cluster_embeddings`) holds rows
referencing the cluster, EVEN when the UPDATE doesn't change the
primary key. This is a known DuckDB limitation documented in the
"Foreign keys" section of their constraints guide.

The workaround `_persist` uses:

1. Capture `cluster_articles` member IDs into a Python list.
2. Capture `cluster_embeddings` row (centroid + embedded_at) if present.
3. `DELETE FROM cluster_articles WHERE cluster_id = ?`
4. `DELETE FROM cluster_embeddings WHERE cluster_id = ?`
5. `UPDATE clusters SET ...` (now no referencing rows → succeeds)
6. Re-INSERT cluster_articles rows.
7. Re-INSERT cluster_embeddings row.
8. INSERT INTO promotion_log (... ON CONFLICT (cluster_id) DO UPDATE).

Each statement auto-commits — no explicit `BEGIN`/`COMMIT`. Within an
explicit transaction the FK check sees the not-yet-committed referencing
rows and still fires; auto-commit between statements is what makes the
pattern work. The trade-off: if the process crashes between the DELETE
and the re-INSERT, the cluster temporarily loses its membership. At
v0.1 this is acceptable — the score stage is seconds per cluster, the
operator can manually re-link from the dashboard if needed, and the
clusterer stage's INSERT OR IGNORE pattern won't recover the rows on a
nightly rerun (the cluster already exists, so step 10 doesn't re-insert
its membership). If this becomes a real operational issue, the fix is
to add `ON DELETE CASCADE` or to switch to a separate `cluster_scores`
table that doesn't share its primary key with `clusters`.

### Promotion log idempotence

`promotion_log` has `cluster_id` as PRIMARY KEY (per ADR-006 schema), so
the score stage's `INSERT ... ON CONFLICT (cluster_id) DO UPDATE`
naturally handles reruns: a second pass with the same cluster_id
overwrites the existing row rather than inserting a duplicate.

The `_select_pending` query filters by `final_defcon IS NULL` so a
cluster scored on the first pass is excluded from the second — the
INSERT pattern is defensive belt-and-suspenders.

## ❯ Mathematical / Statistical Details

**Promotion ceiling.** Deterministic decision tree:

```
ceiling(bands) =
  UNTHINKABLE  if any band in PRIMARY_BANDS
  ROUTINE      else if bands ⊆ SOCIAL_BANDS (X / Reddit only)
  ACUTE        else if |non-social bands| ≥ 3 AND |sides| ≥ 2
  MATERIAL     else if |non-social bands| ≥ 2
  ROUTINE      else (single band)
```

Sides are defined by the disjoint partition `{gov: GOV_ALIGNED,
opposition: OPPOSITION, neutral: CENTRIST ∪ INDEPENDENT ∪
INTERNATIONAL}`. PRIMARY and SOCIAL bands belong to no side.

**Confidence tag.**

```
confidence(bands, n) =
  YÜKSEK   if any primary
  YÜKSEK   else if |sides| ≥ 2 AND n ≥ 4
  ORTA     else if |sides| ≥ 2 OR n ≥ 3
  DÜŞÜK    else
```

**Final DEFCON.** `final = min(raw, ceiling)`. The directional gotcha
is worth flagging: in our enum convention lower numbers mean MORE
severe (UNTHINKABLE=0, AMBIENT=5). `min` chooses the more-severe of
the two operands. The natural-language anchors in ADR-005 use phrases
like "the ceiling can only lower the worker's raw score" — "lower"
here means "lower the score number" = "increase severity". The formula
is correct as written; a future ADR amendment may rephrase the
explanation for clarity.

**Bootstrap demotion.** If `cluster.created_at - MIN(pipeline_runs.
started_at) < 7 days`, apply `final = min(final + 1, AMBIENT)` (one
tier less severe; capped).

## ❯ Design Decisions

**`DEFCON` as `IntEnum`, not `StrEnum`.** Per ADR-004 the formula
`min(raw, ceiling)` operates on integer values. `IntEnum` makes the
arithmetic natural without `int(...)` casts in the formula. Display
strings live in `DEFCON_LABEL_TR`.

**Anchors as a single, FILE-PROTECTED constant.** Every consumer
(prompt builder, briefing renderer, dashboard) reads the same dict. No
copy-paste of anchor strings into the prompt template. ADR amendments
to the anchors propagate everywhere automatically.

**`WorkerResponse` is small.** Six fields: defcon, category,
confidence_self, entities, summary, headline. The worker's job is
classification — not summarisation, not framing. The writer stage
(step 14, Trendyol-LLM) does the prose. Keeping the worker output
small reduces hallucination surface and makes the JSON easier to
validate.

**Confidence tag is the cluster's, not the worker's.** The
`Confidence` enum in the cluster row (YÜKSEK / ORTA / DÜŞÜK) is the
deterministic output of the promotion rules. The worker's
`confidence_self` (high/medium/low) is a separate signal that may be
useful in step 12 (arc-linking) for tie-breaking but is not surfaced
in the briefing.

**Fallback over crash.** If the worker returns malformed JSON
`max_retries + 1` times, the classifier emits a canned
`UNCLASSIFIED + AMBIENT + low` response rather than raising. The
cluster shows up in the briefing, demoted, with the SİSTEM LOG noting
the score fallback. ADR-012 requires degraded operation over hard
failures.

**Bootstrap window is data-defined.** `MIN(pipeline_runs.started_at)`
is the bootstrap origin — no config flag, no separate state. The
operator can effectively "exit bootstrap early" by manually inserting
an earlier pipeline_runs row (or "extend bootstrap" by deleting the
early rows), but no config knob is needed.

**`promotion_log.cluster_id` is the PK.** ADR-006 schema. UPSERT
gives idempotent reruns. The alternative (auto-increment id) would
require extra "did this cluster already get an audit row?" logic.

**No `min` vs `max` second-guessing of ADR-005.** The formula
documented there is `min(raw, ceiling)`. The implementation uses
`min`. The natural-language explanation is awkward (see the gotcha
note in the math section). If ADR amendment changes the formula
later, it's a one-line edit to `final_defcon` in `promotion.py`.

## ❯ Operator caveats

- **Worker prompt updates.** Editing the prompt template in
  `prompt.py` does NOT require an ADR amendment (the prompt is
  operational, not a contract). But major restructuring should be
  validated by an A/B against a held-out cluster set before shipping.
- **DuckDB FK workaround latency.** The DELETE + UPDATE + re-INSERT
  pattern adds 2 extra writes per cluster. At ~100 clusters per night
  this is sub-second. If the cluster count grows past ~10k per night
  the pattern's overhead might matter; revisit.
- **The min/max gotcha.** If a future operator wants to read ADR-005
  and the formula's natural-language description doesn't match their
  intuition, they should NOT change the formula. The formula is
  correct; the prose is just confusing. Direct the operator to this
  doc's math section.
- **UNTHINKABLE doesn't auto-land.** `DEFCON_REQUIRES_OVERRIDE`
  contains UNTHINKABLE. The writer stage (step 14) and dashboard
  (step 19) check this set before serialising a DEFCON 0 to the
  briefing. The classifier itself can WRITE final_defcon = 0 to a
  cluster row, but the row won't surface in the briefing without a
  manual operator override.
- **Vocabulary refresh.** `DEFCON_ANCHORS` is a snapshot of Turkish
  political/economic history as of 2026-05. Major new events (a
  successful coup attempt, a presidential assassination) would
  warrant an ADR amendment extending the anchors. The operator
  reviews quarterly.

## ❯ Same-day formula correction · ADR-005 min → max

Same-day formula correction: the `min(raw, ceiling)` implementation in
step 11 faithfully encoded ADR-005 as written, but the formula was a
latent bug · the ceiling did not act as a severity cap. Discovered
during the ADR-005 prose amendment when worked examples traced through
the three semantic intents (X cap · single-band cap · primary
non-auto-promotion). Formula corrected to `max(raw, ceiling)`. All four
`TestFinalDefcon` assertions and the relevant `test_classifier`
assertions updated. Implementation now matches ADR semantic intent.

The walk-through that exposed the bug:

| raw | ceiling | `min` result | `max` result | ADR intent |
|---|---|---|---|---|
| AMBIENT (5) | ROUTINE (4) — X-only | ROUTINE (4) | AMBIENT (5) | AMBIENT — cap shouldn't escalate noise |
| ROUTINE (4) | UNTHINKABLE (0) — primary | UNTHINKABLE (0) | ROUTINE (4) | ROUTINE — primary doesn't auto-promote |
| SEVERE (2) | ACUTE (1) — 3 bands 2 sides | ACUTE (1) | SEVERE (2) | SEVERE — ceiling permits but doesn't force |
| SEVERE (2) | ROUTINE (4) — single band | SEVERE (2) | ROUTINE (4) | ROUTINE — over-rated, cap applies |

`max` produces the right column for every case. The original prose
("ceiling can only lower the worker's raw score") had been the operator's
proxy for "ceiling caps severity" — but with min, the ceiling could
actually ESCALATE severity (case 3) or contradict its own X-only cap
(case 1). The formula was always supposed to be max; min was a typo
that survived because the test suite validated the formula against
itself, not against the ADR's semantic intent.
