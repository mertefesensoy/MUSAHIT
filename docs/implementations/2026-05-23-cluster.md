# Implementation: Cluster stage · `musahit/cluster/`

**Date** · 2026-05-23
**Author** · Claude Code (Mert Efe Şensoy directing)
**ADR refs** · ADR-001 · ADR-002 · ADR-006 · ADR-008

---

## ❯ Problem / Motivation

Build step 10 of 20. The normalize stage left a populated `articles`
table; the briefing needs to *collapse* those into clusters of "the same
news event being reported by multiple outlets" so the operator sees
**events**, not **outlet duplicates**. Per ADR-008 the cluster row is the
unit that gets a DEFCON score, joins an arc, and lands in the briefing.

Three properties drive the design:

1. **Local, single-machine, single-pass.** ADR-001 forbids long-running
   services. The clustering runs once per night, in-memory, against the
   subset of `articles` that landed in this run.
2. **Cross-outlet, cross-band collapse.** ADR-008 + ADR-005: a cluster
   must aggregate sources from different bands so the promotion logic
   has something to evaluate. We track `bands_present` as the union of
   member sources' bands.
3. **First Ollama-dependent stage.** bge-m3 produces the 1024-dim
   embeddings per ADR-002. Tests must NEVER call real Ollama — the
   `EmbeddingClient` Protocol + `FakeEmbeddingClient` give us a
   network-free path that subsequent Ollama stages (score, arc-link,
   write) will reuse with the same pattern.

## ❯ Implementation Approach

### `EmbeddingClient` Protocol + two implementations

```python
class EmbeddingClient(Protocol):
    async def embed(self, texts: list[str]) -> list[list[float]]: ...
```

Production implementation: `OllamaEmbeddingClient` POSTs to
`/api/embed` (Ollama v0.1.30+) with `{"model": "bge-m3", "input": [...]}`
and returns the matching embeddings. Batching is internal (default 50)
so a 500-article night becomes ~10 HTTP calls. Failures bubble up.

Test implementation: `FakeEmbeddingClient` produces a deterministic
bag-of-words embedding — each whitespace token hashes (MD5, non-secure
context) to a 1024-bucket dimension; weights accumulate; the vector is
L2-normalised. Properties:

* Deterministic — identical text → identical vector.
* Similarity preservation — texts sharing 80% tokens produce cosine
  ≥ 0.8; disjoint texts produce cosine < 0.5 (typically near 0).
* 1024-dim — matches the production embedder so the same threshold,
  same DB schema, same cosine arithmetic apply.

**This is the Ollama-test pattern.** Downstream Ollama-using stages
(score, arc-link, write) should each define their own `FakeXClient`
following the same shape: deterministic outputs, structural properties
that real model outputs would also exhibit (so tests pin the contract
without pinning the model), constructor-injected. Do not introduce
"if testing then ..." branches in the production code.

### Pure cosine + centroid (no numpy)

```python
def cosine_similarity(a, b) -> float:
    dot, norm_a, norm_b = 0.0, 0.0, 0.0
    for x, y in zip(a, b):
        dot += x * y
        norm_a += x * x
        norm_b += y * y
    return dot / (math.sqrt(norm_a) * math.sqrt(norm_b))
```

At our volume — ≤500 vectors × 1024 dimensions per night — the pure-
Python implementation runs in tens of milliseconds total. Adding numpy
would bring in ~25 MB of native code for a single-machine project; the
trade-off doesn't pay off until vector counts grow by 10×. Localised
swap if profiling later disagrees.

Zero-norm vectors return 0.0 cosine (the convention for empty token
bags; no `ZeroDivisionError` leak to the caller).

### Clusterer flow

```text
run(run_id):
  1. SELECT articles JOIN sources JOIN ingest_log LEFT JOIN article_embeddings
     where this run's articles lack an embedding.
  2. Drop articles with word_count < 10 (skipped_empty).
  3. Embed the rest in one call to embedder.embed(texts).
  4. INSERT OR IGNORE into article_embeddings (PK = article_id).
  5. Partition by language. Each unique language value = its own bucket;
     "tr", "en", "unknown" are the common ones but the code partitions
     by exact value so "de", "fr", etc. each get their own bucket.
  6. For each bucket, sort by published_at ascending; greedy single-pass:
       - For each article, find the in-window cluster with the highest
         cosine ≥ threshold.
       - If found → join (update centroid, last_update_at, bands).
       - Else if word_count ≥ 30 → seed a new cluster.
       - Else → skip (headline-only with no match).
  7. INSERT clusters + cluster_articles + cluster_embeddings.
  8. Append "cluster" to pipeline_runs.stages_done; add counts.
```

### Greedy single-pass — order-dependence

The pass is sensitive to order. We sort by `published_at` ascending so
the EARLIEST article in a topic seeds the cluster, later articles
attach. A re-run with the same input produces the same MEMBERSHIP. The
cluster ID's `NNNN` suffix may shift if the daily counter has changed
(e.g., a previous partial run created some clusters), but
which-article-with-which-other-article is stable.

The build-plan tripwire explicitly accepts ID drift while requiring
membership invariance. The integration tests pin this — `TestIdempotence`
re-runs and asserts the cluster count stays at 1.

### Cosine threshold: 0.7 starting guess

Empirical for bge-m3 + news-like Turkish text. The operator should
tune in week 1: too-low produces over-clustering (unrelated stories
merge); too-high produces under-clustering (the same event splits
across outlets). The threshold is a constructor argument, not a code
constant — `config.toml` can set it without touching this module.

(The default lives in `DEFAULT_COSINE_THRESHOLD = 0.7`; future
`config.toml` integration will read from there. For now the operator
edits the constant if needed.)

### Filter thresholds

| word_count | Embedded? | Can join? | Can seed cluster? |
|---|---|---|---|
| < 10 | No | No | No (skipped entirely) |
| 10 – 29 | Yes | Yes (headline-only) | No |
| ≥ 30 | Yes | Yes | Yes |

The headline-only rule matters because Resmî Gazete decrees and Reddit
link-posts with no comments often have very short bodies — they
should attach to a cluster when content matches but not seed clusters
on their own (the briefing would otherwise list dozens of single-
sentence headlines as if each were a distinct event).

`< 10` is the "no usable signal" floor. Reddit empty-body posts and
fragmentary RSS items fall here; saving an embedding is a waste.

### Language partitioning

Articles in different languages never cluster together. This is a
strict partition by the `articles.language` value the normalize stage
wrote. Practical implications:

* `"unknown"` (langdetect's <20-char or LangDetectException output)
  forms its own bucket. Articles with `"unknown"` cluster only with
  other `"unknown"` articles. This is correct because we cannot make
  cross-language similarity claims about text we couldn't identify.
* Tracking by exact value rather than a fixed allow-list means an
  unexpected language code (`"de"`, `"fr"`) gets its own bucket
  rather than forcing into one of `tr/en/unknown`.

### `bands_present` is the source bands' union

The cluster's `bands_present` JSON array is the deduplicated, sorted
list of every member article's `sources.band`. This is what the
promotion stage (step 13) reads to compute `ceiling_defcon` per
ADR-005.

### `pipeline_runs.stages_done` lifecycle (continued)

After cluster runs: `stages_done = ["ingest", "normalize", "cluster"]`.
Counts: `{"articles": N, "articles_normalized": M,
"clusters_new": X, "clusters_joins": Y}`. The status stays RUNNING;
the writer stage marks COMPLETED at the end of the full pipeline.

## ❯ Mathematical / Statistical Details

**Cosine similarity** for unit-norm vectors collapses to the dot product
since both `||a|| = ||b|| = 1`. Our FakeEmbeddingClient L2-normalises
its output, and bge-m3 outputs are L2-normalised by Ollama's
embedding endpoint. We keep the general formula (computing norms
inline) so the implementation works correctly on un-normalised inputs
too — important for the `compute_centroid` output, which is the mean
of unit vectors and is generally NOT unit-norm itself.

**Greedy single-pass clustering** is O(N × K) where N is articles in a
bucket and K is clusters in that bucket. At 500 articles per night
across 3 language buckets, K stays under ~50 per bucket and the total
work is bounded at 75k cosine computations × 1024 dimensions = ~77M
ops. Pure-Python runs this in roughly 1–2 seconds.

The single-pass choice trades optimality (full HDBSCAN-style
clustering would do better at boundaries) for simplicity, determinism,
and explainability. The operator can audit any cluster decision by
re-running the pass with the same input.

## ❯ Design Decisions

**Pure-Python cosine + centroid (no numpy).** Volume doesn't justify
the dependency. Localised swap if profiling later disagrees.

**Embedder owns batching.** Caller hands the full list; the embedder
internally slices into batch-sized chunks. Keeps the orchestrator's
code linear.

**FakeEmbeddingClient is bag-of-words.** A token-overlap-preserving
fake is the cheapest implementation that gives every property we
need: determinism, similarity preservation, the right dimension. A
random-but-seeded fake would give determinism but not similarity
preservation; that would force every test to assert on the same
specific cluster IDs the random generator produced — brittle and
opaque.

**Language partitioning is strict, not soft.** `"tr"` articles never
cluster with `"en"` articles, even if their token bags happen to
overlap (loan words, Turkish names in English text). This matches
human intuition for a news briefing: the operator does not want a
Turkish headline collapsed with a foreign-language headline, even
when they're about the same event.

**Filter thresholds (`< 10` skip, `< 30` no-seed) live in the module.**
These are operational knobs, not architectural decisions. Moving them
to `config.toml` is a follow-up if the operator finds the defaults
wrong. Constants over inline magic numbers; that's enough discipline
for v0.1.

**Cluster IDs share a global daily counter.** All four language
buckets draw from one `NNNN` sequence per day so cluster IDs are
sortable in cluster-creation order regardless of language. This makes
the dashboard's day-view easier to scan.

**`article_embeddings.embedded_at` is the run's start time.** All
embeddings in a single Clusterer.run() get the same `embedded_at`
timestamp. This makes "find all embeddings produced by this run"
trivial via `embedded_at = X`; the alternative (per-embedding
microsecond timestamps) provides no additional signal at our volume.

**No HNSW reliance for in-stage matching.** The HNSW index exists in
the schema (per ADR-006) but it's used by *downstream* queries (arc-
linking, retrospective search), not by THIS stage's matching. In-pass
matching iterates the in-memory `language_clusters` list — fast at our
K values, no DuckDB round-trip per candidate.

## ❯ Operator caveats

- **Threshold tuning in week 1.** Watch for over-clustering (unrelated
  stories merge) and under-clustering (same event splits across
  outlets). Bump `DEFAULT_COSINE_THRESHOLD` in 0.05 increments based
  on the dashboard's cluster-detail view.
- **Greedy is order-dependent.** Sorting by `published_at` makes
  membership stable for a given input, but a delayed straggler post
  (an article ingested with `published_at` in the past) will join
  whichever cluster the SORT puts it next to. If this turns out to
  matter, the right fix is a second-pass refinement, not "make it
  not greedy" — single-pass is the architecturally simpler choice.
- **bge-m3 needs to be already pulled.** Production assumes Ollama
  has `bge-m3` already pulled via `ollama pull bge-m3`. First run
  on a fresh machine will hang waiting for the model download
  unless pre-pulled. Document in operator runbook (step 20).
- **Embedding failure = empty pass.** If the embedder raises (Ollama
  process down, model not pulled), the Clusterer logs and continues
  with zero clusters. The next stage sees no new clusters; the
  briefing surfaces this as a SİSTEM LOG entry.
- **Cluster IDs may shift on rerun.** If you manually create clusters
  via the dashboard between runs, the daily counter advances, and a
  re-run of the same day's pass will produce different IDs (but the
  same MEMBERSHIP). Don't pin URLs on cluster IDs in external docs.
