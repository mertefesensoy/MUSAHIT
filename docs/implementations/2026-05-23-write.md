# Implementation: Briefing writer · `musahit/writer/`

**Date** · 2026-05-23
**Author** · Claude Code (Mert Efe Şensoy directing)
**ADR refs** · ADR-002 (amended 2026-05-23) · ADR-009 · ADR-012

---

## ❯ Problem / Motivation

Build step 13 of 20. The pipeline now has scored clusters and linked
arcs; this stage turns them into the operator-facing artefact — the
nightly Turkish briefing. The briefing is the *single thing* the
operator reads with morning coffee. Its structure shapes the
operator's mental model of what happened in Turkey; its prose quality
determines whether the operator gets useful signal or skims past clunky
text.

Three properties drive the design:

1. **Idiomatic Turkish prose.** The writer model is
   Trendyol-LLM 7B chat v1.8 (the ADR-002 amendment from
   2026-05-23 — v4 doesn't exist publicly). Pulled directly from
   Ollama Hub as `serkandyck/trendyol-llm-7b-chat-v1.8-gguf` with a
   single `ollama pull`.
2. **Structural reliability.** ADR-009 specifies eight required
   top-level sections with explicit `❯` markers. Downstream parsers
   (TTS scope extractor, dashboard renderer) key off these markers.
   The writer is constrained by a template validator; structurally
   bad output triggers a retry with the validator's errors fed back
   into the prompt.
3. **The briefing always ships.** Per ADR-012 § Stage 6, if the writer
   model produces malformed output after 3 retries, the pipeline
   switches to a deterministic Python-rendered fallback. The
   operator gets a briefing — just uglier than the LLM-generated
   one — never an empty inbox.

## ❯ Implementation Approach

### Two-step retry + fallback

```text
build_payload(conn, run_id)
        │
        ▼
build_writer_prompt(payload)      ← initial prompt
        │
        ▼
   ┌──── attempt loop (up to max_retries + 1) ────────────┐
   │                                                       │
   │   llm.generate(prompt) ──► raw markdown               │
   │            │                                          │
   │            ▼                                          │
   │   validate_briefing_markdown(raw) ─► []? ──► return   │
   │            │                                          │
   │           errors                                      │
   │            │                                          │
   │            ▼                                          │
   │   prompt := base_prompt + "validator complained: …"   │
   │   (retry with feedback)                               │
   │                                                       │
   └──── if all retries fail ────────────────────────────► render_fallback_briefing(payload)
                                                            │
                                                            ▼
                                          briefing.md (writes via Python renderer)
```

The retry prompt appends the validator's specific error lines so the
model has a chance to self-correct. This pattern is documented in
ADR-009 § Negative ("if invalid, the writer is re-prompted with the
specific error").

If the LLM itself raises (Ollama down, timeout, etc.) the orchestrator
also counts that as a failed attempt and either retries or falls
through. The fallback path always succeeds because the Python renderer
is deterministic and guaranteed validator-clean (the renderer pulls
section markers from `template.TEMPLATE_SECTIONS`, same source as the
validator).

### Template validator approach

```python
def validate_briefing_markdown(text: str) -> list[str]:
    # 1) Title is the first non-blank line.
    # 2) Every top-level `## ` line is checked in order against TEMPLATE_SECTIONS.
    # 3) Missing required sections → error.
    # 4) Out-of-order required sections → error.
    # 5) Any `## ` line not in TEMPLATE_SECTIONS → error.
    # 6) Required sections without `❯` prefix → error.
    # No content-shape checks; the LLM is free on prose.
```

The validator is tight on **structure** and loose on **content**.
H3 (`###`) subsections are content — any number, any text. H2 (`## `)
markers are structure — exact text match required. This split lets the
LLM use as many `### Item Title` blocks as the day warrants without
ever drifting on the eight section anchors that TTS / dashboard /
audit need.

**Tripwire check (validator too strict):** the validator's only string
comparison is the exact marker line. It does not enforce whitespace
between sections, paragraph length, or any prose pattern. If real LLM
output ever fails on a trivial whitespace difference, that's an
implementation bug, not the validator's design — raise it; don't
loosen the marker comparison.

### `BriefingPayload` shape

```python
@dataclass(frozen=True)
class BriefingPayload:
    date: date
    run_id: str
    clusters_by_defcon: dict[int, list[ClusterView]]
    open_arc_updates: list[ArcView]
    resolved_arcs: list[ArcView]
    peak_defcon: int
    cluster_count: int
    arc_count: int
    open_arc_count: int
    ambient_count: int
    failed_sources: list[FailedSource]
    stages_done: list[str]
```

The payload is read once at the start of `Briefer.run` and reused
across all LLM attempts (the retry prompt sees the same data; only
the appended error feedback changes). All sub-views are frozen
dataclasses — easy to test against fixture payloads without the DB.

`clusters_by_defcon` keys by integer DEFCON. The prompt builder and
fallback both bucket via the constants
`BUCKET_PRIORITY=(0,1,2)`, `BUCKET_MATERIAL=(3,)`, `BUCKET_ROUTINE=(4,)`,
`BUCKET_AMBIENT=(5,)` to map to the four content sections.

### Prompt size estimate

Worst-case input size:

| Component | Tokens |
|---|---|
| System role + DEFCON schema + template skeleton + discipline rules | ~3K |
| 50 priority/material clusters @ ~150 tokens | ~7.5K |
| 20 routine clusters @ ~50 tokens | ~1K |
| 10 open arcs + 5 resolved arcs @ ~100 tokens | ~1.5K |
| System log + failed-source list | ~200 |
| **Total input** | **~13K** |

Trendyol-LLM 7B v1.8 has a 32K-token context window. 13K input leaves
~19K for the model's response (the briefing markdown itself, which is
typically 2-4K tokens). Headroom is comfortable; no sectioned
generation needed at v0.1 volumes.

A pytest guard (`test_prompt_well_under_32k_chars`) asserts a fixture
prompt comes in under ~16K characters (~4K tokens) — a sanity net
against accidentally dumping raw article bodies into the prompt.

### Per-date idempotence

The `briefings` table has `date` as PRIMARY KEY (ADR-006). The briefer
uses `INSERT … ON CONFLICT (date) DO UPDATE` so re-running for the same
date overwrites the row rather than failing. The markdown file is also
overwritten on disk. This matches the operator's expectation that the
briefing is *per-date*, not *per-run* — manual reruns refresh the
artefact rather than duplicating it.

### ADR-002 amendment reference

Per the 2026-05-23 ADR-002 amendment, the writer model is
`serkandyck/trendyol-llm-7b-chat-v1.8-gguf`. The Briefer reads this
from `Settings.writer_model` (defaulted in `config.toml`); tests inject
`FakeLlmClient` so the real model is never pulled or called in CI.

## ❯ Design Decisions

**Fallback over crash.** Per ADR-012, the briefing always ships. The
Python fallback's output is structurally identical to the LLM's
expected output (same markers, same order) so downstream stages (TTS,
dashboard) handle it without special-casing.

**Validator is structure-only.** Content checks (prose length,
attribution discipline, quote count) would mistakenly reject valid
output on trivial differences. Discipline lives in the prompt's
"KURALLAR" block; structural sanity lives in the validator. If
discipline drifts, the operator notices and tunes the prompt — that's
a prompt edit, not a code edit.

**Retry prompt feeds errors back.** The retry prompt includes
"ÖNCEKİ DENEMEDE ŞABLON DOĞRULAYICI HATA VERDİ: …" with up to 5 error
strings. The model has a fair chance to self-correct. The retry
prompt is a string concatenation — no re-querying the DB, no new
clusters / arcs surfaced mid-retry.

**Prompt uses the project's DEFCON anchors directly.** No copy-paste
of ladder definitions — `prompt.py` reads `DEFCON_ANCHORS` from
`musahit.score.defcon` (the FILE-PROTECTED source of truth). Any
future ADR-004 amendment propagates automatically.

**Pure-data payload.** `BriefingPayload` and its sub-views are frozen
dataclasses with no DB handles. The fallback renderer and the prompt
builder both consume the same shape, which means tests can drive
either with fixture payloads.

**Per-date idempotence, not per-run.** A pipeline re-run for the same
date refreshes the briefing rather than producing a second copy. The
`pipeline_runs` table tracks the *run*; the `briefings` table tracks
the *date*. Per ADR-006.

**Briefer takes a `briefings_root` path argument.** Production passes
`Settings.briefings_dir` (defaults to `briefings/`). Tests pass
`tmp_path / "briefings"`. No environment-aware path-resolution; the
caller decides.

**FakeLlmClient's per-prompt attempt counter — closure-based test counter.**
The score-stage `FakeLlmClient` tracks `attempt` per unique prompt key.
The writer's retry mechanism *changes* the prompt each retry (it
appends validator errors), so the per-prompt-key counter resets to 0
on each retry. The writer integration tests therefore use a
closure-scoped counter in the responder when they want to script
behaviour across attempts. This is a test-only concern; production
code doesn't read the counter.

## ❯ Operator caveats

- **First-night writer tuning.** Trendyol-LLM v1.8 has not been
  battle-tested on this prompt shape. Expect the first 1-2 nights to
  reveal prompt-engineering tweaks (probably in `DISCIPLINE_RULES`).
  The validator gives a deterministic feedback signal; if a specific
  validator error appears repeatedly, that's a prompt issue, not a
  code issue.
- **The fallback is structurally fine but stylistically utilitarian.**
  If a night uses the fallback (visible in `pipeline_runs.counts.
  writer_used_fallback`), the operator should investigate — usually
  Ollama wasn't running or Trendyol-LLM wasn't pulled. Re-run the
  step manually to recover.
- **`audio_path` is NULL after this stage.** Step 14 (TTS) populates
  it. The dashboard's banner state machine treats NULL audio_path as
  "audio pending" until that stage runs.
- **HTML path is a planned location, not a real file.** The dashboard
  (step 19) renders the markdown into HTML. Until then, `html_path`
  in the `briefings` row points to a file that doesn't yet exist on
  disk; the field is a forward declaration.
- **Markdown file overwrites on rerun.** Operator-side workflows that
  cache the briefing.md by file mtime should rerun their workflow
  after a manual writer re-run. Per-date idempotence is documented
  but worth flagging in the operator runbook (step 20).
- **Per-prompt-attempt counter quirk** (FakeLlmClient). Test-only.
  Documented in the design decisions above. Production code does not
  read the counter; the briefer's own loop variable tracks attempts.
