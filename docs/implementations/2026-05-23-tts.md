# Implementation: TTS · Piper audio synthesis (step 14)

**Date** · 2026-05-23
**Author** · Mert Efe Şensoy
**ADR refs** · ADR-009 · ADR-010 (amended 2026-05-23) · ADR-012

---

## ❯ Problem / Motivation

The writer (step 13) produces `briefing.md` containing the day's full
structured Turkish briefing. ADR-009 § TTS scope specifies that a
subset of the briefing — header + DEFCON 1-2 + DEFCON 3 summaries +
AÇIK GELİŞMELER + a closing line — must also ship as an MP3 audio
track for the operator to listen to while drinking morning coffee.
The audio file is referenced by `briefings.audio_path` and served by
the dashboard at `/briefing/audio/today`.

ADR-010 (amended 2026-05-23) pins the synthesis engine: **Piper TTS**
via the Python package `piper-tts` (`from piper import PiperVoice`).
The original ADR text described a subprocess invocation of the
archived `rhasspy/piper` binary; the amendment moved to the Python
API as the canonical integration path and explicitly forbids silently
reverting to subprocess.

ADR-012 § Stage 7 TTS requires that the pipeline NEVER abort on a TTS
failure. Even when Piper crashes, when ffmpeg is missing, or when the
briefing markdown is malformed, an MP3 file MUST be written and
`briefings.audio_path` set so the dashboard's `<audio>` element loads
something. The fallback is a 1-second silent placeholder.

## ❯ Implementation Approach

### Pipeline shape

```
briefing.md
  ├─→ extract_voiced_briefing(md) → VoicedBriefing(header, defcon_priority, defcon_material, open_arcs, closing)
  ├─→ for each non-empty chunk: preprocess_for_tts(chunk)
  ├─→ for each cleaned chunk: piper.synthesize(text) → WAV bytes
  ├─→ interleave WAVs with generate_tick_tone() (tick between, not at edges)
  ├─→ concatenate_wavs([…])
  ├─→ wav_to_mp3(combined)
  └─→ write briefings/YYYY/MM/DD/briefing.mp3 + UPDATE briefings.audio_path
```

### Extractor parsing rules

The parser walks the briefing markdown line by line. When it sees a
line that matches one of the ADR-009 `## ❯ …` markers (compared by
exact equality against the constants in `extractor.py`), it switches
the current bucket. Lines before any marker go into a synthetic
`__header__` bucket. Marker lines themselves are NOT included in the
bucket text — only the section bodies are kept, so the synthesiser
doesn't end up reading the marker line aloud.

For the DEFCON 3 section specifically, `**Kaynaklar** · …` lines are
stripped after bucketing: per ADR-009 § TTS scope the source list is
visual-only.

The closing line `"DEFCON 4 ve sonrası dashboard'da görüntülenebilir."`
is a constant; the extractor doesn't read it from the markdown.

Chunk order (returned by `VoicedBriefing.chunks()`):

1. Header
2. DEFCON 1-2 ÖNCELİKLİ
3. DEFCON 3 MATERYAL (sans Kaynaklar)
4. AÇIK GELİŞMELER · DEVAM EDEN TAKİP
5. Closing line

Empty sections are dropped from the chunk list so we don't waste a
Piper call on an empty string (Piper would synthesize silence which
is wasted time *and* produces awkward tick-tick-tick transitions with
no speech between).

### Preprocessor pipeline

The function is a pure transformation `str → str`. Order matters:

1. Drop `**Kaynaklar** · …` lines (regex on a multiline span; bold
   markers may or may not bracket "Kaynaklar")
2. Strip markdown formatting — headers (`# `, `## `, `### `, `#### `,
   etc.), bold (`**x**` → `x`), italic (`*x*` → `x`, with negative
   lookahead so bold isn't half-stripped), links (`[label](url)` →
   `label`), horizontal rules (`---`), arrow markers (`❯`)
3. Replace `DEFCON N` numerals with `DEFCON {Bir/İki/Üç/Dört/Beş}` so
   the voice reads them in Turkish rather than (sometimes) English
4. Expand abbreviations (TCMB → "Te Ce Me Be", etc.) — whole-word
   match, case-sensitive (MÜŞAHİT's output normalises acronyms to
   uppercase, so a case-sensitive match is both safer and faster than
   re-matching mixed case)
5. Collapse runs of 3+ blank lines to a single blank line — Piper's
   sentence-boundary detection handles natural pauses; multiple blank
   lines produce no useful prosodic effect

### Piper integration (ADR-010 amended)

```python
from piper import PiperVoice  # lazy import in __init__

class PiperPythonClient:
    def __init__(self, voice_path: Path, *, timeout_seconds=60.0):
        if not voice_path.exists():
            raise FileNotFoundError(...)
        self._voice = PiperVoice.load(str(voice_path))   # ~300ms one-time
        self._timeout_seconds = timeout_seconds

    async def synthesize(self, text: str) -> bytes:
        return await asyncio.wait_for(
            asyncio.to_thread(self._synthesize_sync, text),
            timeout=self._timeout_seconds,
        )

    def _synthesize_sync(self, text: str) -> bytes:
        buf = io.BytesIO()
        with wave.open(buf, "wb") as wav_file:
            self._voice.synthesize_wav(text, wav_file)
        return buf.getvalue()
```

API surface note: ADR-010's amendment example showed
`voice.synthesize(text, f)`. The current `piper-tts==1.4.2` API actually
exposes:

- `voice.synthesize(text)` → `Iterable[AudioChunk]` (raw frames per sentence)
- `voice.synthesize_wav(text, wav_file)` → writes a complete WAV through `wave.Wave_write`

We use `synthesize_wav` — it handles the format-setting on first chunk
and concatenates frames into a single WAV body, which is exactly what
the downstream `concatenate_wavs` and `wav_to_mp3` pipeline expects.
No subprocess. No binary in PATH. The Python API is the canonical
integration path.

### Transition tones

200 ms of 80 Hz sine wave at 25 % amplitude, with 5 ms linear
attack/decay envelope to soften clicks at boundaries. Generated via
pure Python (`math.sin` + `struct.pack` + `wave.open`); no numpy
dependency. Cached after first call so the synthesiser's
interleave loop reuses one buffer rather than regenerating per
section.

Format matches the Piper voice: 22050 Hz, 16-bit signed, mono. This
lets `concatenate_wavs` glue Piper output and tick WAVs together
without resampling.

### WAV concatenation

Pure-stdlib `wave` round-trip: read each input via
`wave.open(BytesIO(b), "rb")`, validate that framerate / sampwidth /
nchannels match across all inputs (mismatch raises `ValueError` — the
TTS pipeline guarantees uniform format upstream so a mismatch is a
bug to surface, not silently resample around), accumulate frame
bytes, write a single output WAV. Empty input list yields a
zero-frame header (keeps the call-site simple).

### MP3 encoding

`pydub.AudioSegment.from_wav(BytesIO(wav)).export(..., format="mp3",
bitrate="128k")`. Pydub shells out to ffmpeg under the hood, so this
function requires ffmpeg in PATH at runtime. On a machine without
ffmpeg, the function raises; the synthesiser catches the broad
`Exception` and falls back to the silent placeholder. The test for
`wav_to_mp3` itself uses `pytest.skipif(not shutil.which("ffmpeg"))`
so the suite stays green on machines without ffmpeg.

Python 3.13 removed the stdlib `audioop` module that pydub depends
on. The `audioop-lts` backport is added as a conditional dep
(`python_version >= '3.13'`) so pydub imports cleanly on the current
operator install.

### Always-ships failure handling

The synthesiser wraps the entire synthesis chain (extract → preprocess
→ Piper → concatenate → MP3) in a single `try` block. ANY exception
falls through to a placeholder path:

```python
mp3_bytes = silent_placeholder_mp3()  # 1s of WAV-bytes-in-mp3-file
used_placeholder = True
reason = f"{type(exc).__name__}: {exc}"
```

The placeholder is then written to `briefing.mp3` and
`briefings.audio_path` is updated regardless. `stages_done` always
appends `tts` and `counts["tts_used_placeholder"]` records whether
the placeholder fired. This honours ADR-012's "briefing always ships"
invariant.

#### Placeholder format

The placeholder is 22 050 frames (1 s @ 22050 Hz) of int16 zeros,
wrapped in a valid WAV header, written into a `.mp3` extension. HTML5
`<audio>` content-sniffs the bytes rather than trusting the
extension, so the dashboard's audio element plays the WAV silently
when the file fires. Full MP3 fidelity only happens on the healthy
path; the failure path uses pure stdlib so it can never itself fail
on a missing ffmpeg.

### Idempotence

The synthesiser is keyed by the most recent `briefings.date` row
(`ORDER BY generated_at DESC LIMIT 1`). Multiple runs on the same
date overwrite `briefing.mp3` and update `briefings.audio_path` in
place — mirroring the writer's per-date idempotence. No duplicate row
is created.

### Dependency injection seams

- **Piper**: constructor takes a `PiperClient` (Protocol). Tests
  inject `FakePiper` (returns deterministic silent WAV bytes per
  call) or `FailingPiper` (raises on every call).
- **MP3 encoder**: constructor takes an optional `mp3_encoder:
  Callable[[bytes], bytes]` keyword, defaulting to `wav_to_mp3`. Tests
  inject a tiny fake that returns `b"FAKE_MP3:" + wav[:8]` so the
  happy-path synthesiser test runs on machines without ffmpeg. The
  goal-spec constructor signature `(db, piper, briefings_root)` is
  preserved as the positional contract; the encoder is a keyword-
  only escape hatch.

## ❯ Mathematical / Statistical Details

### Tick-tone signal

Each sample `s_i` of the 200 ms 80 Hz transition tick is computed as:

```
s_i = A · env(i) · sin( 2π · f · i / sr )
```

where:

- `A = 8000` (≈ 25 % of int16 full-scale 32767)
- `f = 80 Hz`
- `sr = 22050 Hz` (sample rate matching the Piper voice)
- `i ∈ [0, sr × 0.2)` (4410 samples)
- `env(i)` is a piecewise-linear attack/decay envelope:
  - rising 0 → 1 across the first 5 ms (110 samples)
  - holding at 1 in the middle
  - falling 1 → 0 across the last 5 ms

200 ms × 80 Hz = 16 complete sine cycles, so the wave begins and ends
at a zero crossing — combined with the envelope, this eliminates the
click that would otherwise come from a hard cut.

`s_i` is clipped to `[-32768, 32767]` (`_clip16`) and packed as little-
endian int16 (`struct.pack("<h", ...)`) before being written as PCM
frames.

### Briefing chunk-count vs Piper calls

For a nominal briefing covering all four voiced sections plus the
closing line, the synthesiser issues 5 Piper calls. For a degraded
briefing where, say, DEFCON 1-2 is empty, the synthesiser drops that
chunk and issues 4 calls. The transition tick count is always
`max(0, chunks − 1)` since ticks live between chunks, not at the
edges.

## ❯ Design Decisions

### Why a Protocol + DI instead of patching pydub/piper at import time

Same pattern as `LlmClient`, `EmbeddingClient`, `Ingester` in earlier
stages. The Protocol makes the contract explicit, the fakes are
trivially testable, and the production class can change its
internal mechanics (e.g., swap to `synthesize_wav` from the older
`synthesize(file)` shape that the ADR-010 amendment hinted at)
without rippling through test code. Patching `from piper import
PiperVoice` at module load would have left a subtle dependency on
import order that would bite later.

### Why an optional `mp3_encoder` keyword instead of injecting via Protocol

The goal-spec constructor reads `(db, piper, briefings_root)` —
positional, three args. Adding a fourth positional would break that
contract. A keyword-only optional argument honors the spec while
preserving testability on ffmpeg-less machines. The default is
`wav_to_mp3` so production behaviour is unchanged.

### Why concatenate WAVs in Python rather than pipe to ffmpeg

Pure stdlib `wave` round-trip is < 5 ms for the briefing-sized inputs
we see. ffmpeg's startup overhead per pipe invocation is ~30 ms on
Windows. Doing it in Python is faster AND removes a subprocess from
the hot path (the spirit of ADR-010's amendment — though it
technically only restricts Piper, not pydub).

### Why a WAV-bytes-in-.mp3-extension placeholder rather than a hardcoded MP3 frame

The placeholder must satisfy three constraints:

1. Always producible (no external system dep, no codec install)
2. Playable in the HTML5 `<audio>` element
3. Bounded size (no megabytes of zeros)

A 1-second silent WAV at 22050 Hz mono is ~44 KB, generated with
zero dependencies, and is content-sniffed correctly by every modern
browser regardless of the file's extension. A hardcoded MP3 frame
(~417 bytes) would have been smaller and "more correct" but would
have added either (a) a base64-encoded byte literal in the source
(non-obvious and harder to audit) or (b) a runtime MP3-frame
generator (significant complexity for a fallback path that's only
exercised on failure).

The trade-off is documented and visible — operator-facing comments
on `silent_placeholder_mp3()` explain that the file is WAV-bytes
inside an `.mp3` extension. When ffmpeg is healthy (the nominal
case) the real MP3 path runs and the placeholder never fires.

### Alternative considered: skip the briefing.mp3 file entirely on failure

Rejected. The dashboard's `/briefing/audio/today` route would 404, and
the `<audio>` element would render a "broken file" icon. Worse, the
operator might think the pipeline didn't run at all rather than a
single stage failing. The silent placeholder produces an unambiguous
"loaded silent track" UX — combined with the dashboard banner per
ADR-010 § Fallback, the operator gets a clear failure signal.

### Alternative considered: Numpy for tick-tone generation

Rejected. Step 10's centroid math deliberately stayed numpy-free; the
project hasn't pulled numpy in as a dep yet. A 4 400-sample sine wave
is trivially fast in pure Python. Avoiding numpy keeps the install
footprint small and avoids the question of which numpy version to
pin (some scientific-Python wheels are large on Windows).

## ❯ Operator caveats

- **ffmpeg required for real MP3 encoding.** Install via
  `winget install Gyan.FFmpeg` (or equivalent). Without it,
  `wav_to_mp3` raises and the synthesiser writes the silent
  placeholder — the operator should diagnose by checking
  `pipeline_runs.counts.tts_used_placeholder` in the briefing's
  SİSTEM LOG footer.
- **Piper voice ONNX must be present at `piper_voice_path`.** The
  default is `C:/Users/senso/AppData/Local/piper/voices/tr_TR-dfki-medium.onnx`.
  `PiperPythonClient.__init__` raises `FileNotFoundError` with a
  pointer to `scripts/install_windows.ps1` if the file is missing.
- **Audioop on Python 3.13+.** The stdlib `audioop` module was removed
  in Python 3.13. We add `audioop-lts` as a Python-3.13-gated
  dependency so pydub imports cleanly. If the operator ever
  downgrades Python, the conditional dep won't install and pydub
  will use the stdlib version — no extra steps needed.

## ❯ Same-day diagnostic improvements (post-smoke-test, 2026-05-23)

After step 14 landed, the operator ran a smoke test that produced a silent
placeholder MP3. Initial confusion was high — the dashboard would have shown
"audio synthesis failed" but with no operator-visible clue as to why. Root
cause turned out to be trivial: ffmpeg wasn't on PATH for the PowerShell
session that ran Python. Two gaps had been hiding it:

1. **Synthesizer's exception handler swallows the traceback.** The catch-all
   `except Exception` writes the placeholder, calls `log.warning(...)`, and
   continues. When `configure_logging()` hasn't been called (manual
   invocations, smoke tests, ad-hoc reruns), the structured log call is a
   silent no-op and the operator sees nothing.
2. **ffmpeg's absence isn't checked upfront.** Pydub launches ffmpeg via
   `subprocess` without a pre-flight check; on a machine without ffmpeg the
   failure surfaces as `FileNotFoundError [WinError 2]` deep inside pydub's
   `subprocess.Popen` — accurate but opaque to anyone not already suspicious
   of the env.

### What changed

| File | Change |
|---|---|
| `musahit/tts/encoder.py` | Added `shutil` import. New `check_ffmpeg_available()` that calls `shutil.which("ffmpeg")` and raises `RuntimeError("ffmpeg not found on PATH · install via winget install ffmpeg on Windows or apt install ffmpeg on Linux")`. Module-load call wrapped in try/except → `warnings.warn(..., UserWarning)`. `wav_to_mp3` calls the check at the start of its body (strict raise). |
| `musahit/tts/synthesizer.py` | Added `sys` and `traceback` imports. The exception handler now calls `traceback.print_exc(file=sys.stderr)` BEFORE the structured `log.warning(...)` call. Both signals fire — manual invocations see the traceback, production logging picks up the structured event. |
| `tests/test_tts/test_encoder.py` | `TestCheckFfmpegAvailable` × 3: raises with the expected message when `shutil.which` returns None, passes silently when it doesn't, and `wav_to_mp3` short-circuits via the strict check rather than letting pydub's `FileNotFoundError` leak out. |
| `tests/test_tts/test_synthesizer.py` | `test_piper_crash_prints_traceback_to_stderr` uses `capsys` to assert that `Traceback`, the exception type name, and the message all appear on stderr — while keeping the placeholder MP3, `audio_path` update, and `used_placeholder` flag behaviour intact. |

### Design notes

- The module-level UserWarning is once-per-process at import time and is
  visible in the pytest warnings summary on machines without ffmpeg. It does
  not prevent imports; unit tests for the extractor, preprocessor,
  transitions, and Piper Protocol don't actually need ffmpeg and continue to
  pass cleanly.
- The strict raise lives in `wav_to_mp3`'s body so the existing
  `@pytest.mark.skipif(not _ffmpeg_available())` decorator on
  `test_wav_to_mp3_roundtrip` continues to gate that test on a real ffmpeg —
  the skip-decorated test never reaches the strict check.
- The exception-handler's traceback print fires *before* the log call so the
  output ordering is: traceback → structured log → placeholder write. This
  ordering matches what an operator scrolling a terminal sees first
  (traceback at the top of the failure block, then structured event, then
  the placeholder confirmation in the next stage's log).
- These changes are pure diagnostic improvements — no behaviour change in the
  always-ships invariant. Piper crash still produces a silent placeholder;
  ffmpeg missing still produces a silent placeholder; the only difference is
  that the operator now sees *why* on stderr instead of having to guess.

Full suite after the diagnostics landed: **517 passed, 2 skipped** (+4 new
tests). The existing `test_wav_to_mp3_roundtrip` continues to skip when
ffmpeg is unavailable.

### Audio QA: DEFCON pronunciation nudge

Operator audio QA found "DEFCON" was being pronounced with Turkish phoneme
rules (roughly "De-Fe-Kon") instead of the English-style "Def-Kon" that
Turkish speakers familiar with the term actually use. The preprocessor now
respells the TTS-bound text as "Defkon" to nudge Piper toward correct
pronunciation. The written briefing is unchanged — only the text that flows
into `PiperVoice.synthesize_wav` gets the respelling.

| Aspect | Before | After |
|---|---|---|
| Briefing markdown | `DEFCON 2` | `DEFCON 2` (unchanged) |
| TTS-bound text | `DEFCON İki` | `Defkon İki` |
| Piper output | "De-Fe-Kon İki" | "Def-Kon İki" |

Implementation lives in `_defcon_num_repl` (a one-line change from `f"DEFCON
{…}"` to `f"Defkon {…}"`). The Turkish-numeral expansion is unchanged. The
test suite gained per-level assertions
(`TestDefconNumbering::test_defcon_{1..5}_to_{bir,iki,uc,dort,bes}`) plus a
guard against silent regression to the all-caps form
(`test_all_defcon_levels_use_defkon_respelling`) and a guard against false-
positive respelling of bare DEFCON prose
(`test_defcon_with_no_numeral_left_alone`).
