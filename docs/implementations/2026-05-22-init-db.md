## ❯ Problem / Motivation

Build step 2 of 20 per BOOTSTRAP.md. Every pipeline stage (ingest, normalize, cluster, score,
arc-link, write) reads and writes the tables defined in ADR-006. Without an initialized DuckDB
schema there is no persistence layer — no other module can run.

The init must be safe to rerun so the operator can call `python scripts/init_db.py` after every
`git pull`, and so the Task Scheduler install script can call it idempotently.

The VSS extension (vector similarity search, required for HNSW indices) downloads from the
internet at first use. The init must not fail on a network-restricted machine — it should apply
the schema and log a warning about missing HNSW indices.

## ❯ Implementation Approach

### Migration version tracking

A `schema_version` table (created by `ensure_schema_version_table` before any migration scan)
records each applied migration with its version number and a timestamp. On every run:

1. Query `schema_version` for already-applied versions.
2. Glob `scripts/migrations/[0-9][0-9][0-9]_*.sql` and sort lexically (zero-pad makes this
   numerically equivalent).
3. For each file whose parsed version is not in the applied set, run it in a `BEGIN`/`COMMIT`
   transaction and insert a `schema_version` row on success. Rollback + reraise on failure.

The version check is the primary idempotence guard. `CREATE TABLE IF NOT EXISTS` and
`CREATE INDEX IF NOT EXISTS` throughout the SQL are belt-and-suspenders.

### Multi-statement SQL execution

DuckDB Python's `execute()` takes one statement at a time. The SQL file is split on `;`
**after** stripping line comments via `re.sub(r"--[^\n]*", "", sql)`. Stripping comments first
ensures that a semicolon inside a comment (e.g. `-- note; see docs`) does not produce a
spurious empty statement that the parser would reject.

String literals in schema SQL never contain semicolons, so the simple split is safe.

### VSS fallback

```
init_db(load_vss=True)
  │
  ├─ make_connection(db_path, load_vss=False)   ← always opens without VSS
  │                                              ← never lets make_connection hard-fail
  ├─ try_load_vss(conn, log)
  │    └─ _install_and_load_vss(conn)           ← monkeypatchable in tests
  │         ├─ success → returns True
  │         └─ duckdb.Error → log WARN, return False
  │
  ├─ ensure_schema_version_table(conn)           ← unconditional
  ├─ apply_pending_migrations(conn, ...)         ← unconditional
  │
  └─ if vss_loaded:
       create_hnsw_indices(conn, log)
            ├─ for each index: CREATE INDEX IF NOT EXISTS ... USING HNSW
            │    ├─ success → count += 1
            │    └─ duckdb.Error → log WARN, continue next index
            └─ return count
```

Key invariant: the connection is **always** opened with `load_vss=False`. VSS is handled
entirely inside `init_db`, so the try/except in `try_load_vss` is the only guard needed. If
`make_connection(load_vss=True)` were called instead, it would hard-fail before our fallback
logic could run.

### `scripts/init_db.py` sys.path bootstrap

The script inserts the repo root into `sys.path[0]` before any `import musahit`. This makes
`python scripts/init_db.py` work whether or not the package is installed via `pip install -e .`.
Task Scheduler typically sets `sys.path[0]` to the script directory (`scripts/`), not the repo
root, so the bootstrap is essential for the nightly unattended use case.

## ❯ Design Decisions

**Logic in `musahit/common/migrations.py`, not `scripts/init_db.py`**
`scripts/` contains heterogeneous files (PowerShell scripts, XML). Making it a Python package
(by adding `__init__.py`) would be semantically odd. Putting the logic in `musahit/common/`
keeps it importable by tests via `from musahit.common.migrations import init_db`, with the
script being a thin wrapper — the standard pattern for testable CLI tools in Python.

**HNSW indices in Python, not SQL**
Embedding HNSW creation in the SQL migration file would make VSS a hard dependency for the
migration: any `USING HNSW` DDL without VSS loaded raises immediately. Keeping HNSW in Python
allows the per-index soft-fail design (each index in its own try/except) and makes the SQL file
fully VSS-agnostic.

**`make_connection(load_vss=False)` inside `init_db`**
See VSS fallback section above. This prevents `make_connection`'s own VSS logic from running
(which would hard-fail before the `try_load_vss` fallback has a chance to log WARN and continue).

**`IF NOT EXISTS` idempotence caveat**
`CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` are used throughout. This is
idempotent, but it is **not a schema-repair mechanism**: if a previous run left a table with
wrong columns (e.g. a partial migration that modified a column but crashed mid-way), the `IF NOT
EXISTS` on the `CREATE TABLE` will silently skip the table without fixing the columns. Schema
drift detection and repair require a new migration file — they are out of scope for v0.1.

**`_install_and_load_vss` as a separable helper**
The two-line `INSTALL vss; LOAD vss` is extracted into `_install_and_load_vss()` so that tests
can monkeypatch it to simulate a network failure without mocking the entire connection object.
This makes `TestVssLoadFailureIsSoft` precise about what it is testing.
