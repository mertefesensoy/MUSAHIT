## âťŻ Problem / Motivation

Steps 4-14 built each of the seven stages independently â€” ingest,
normalize, cluster, score, arc-link, write, tts â€” each with its own
orchestrator class, its own `run(run_id)` method, its own test rig.
Step 15 wires them into a single nightly run.

The orchestrator's contract is set by ADR-001 Â§ Stage checkpoints +
ADR-007 Â§ Resumability + ADR-012 Â§ Failure isolation by stage:

- **Sequential execution** in canonical order (ingest â†’ normalize â†’
  cluster â†’ score â†’ arc-link â†’ write â†’ tts)
- **Per-stage failure isolation** â€” one stage's exception is logged
  and recorded in `failed_stages`, the next stage runs anyway
- **Resumability** â€” re-running with the same `run_id` skips stages
  already in `stages_done` (Windows Update reboot mid-run recovery)
- **Per-stage timing budget** per ADR-007 â€” soft deadline logged,
  hard timeout at 2Ă— the budget aborts the stage and continues
- **Ollama model lifecycle** per ADR-001 Â§ Single Ollama instance â€”
  load the model each stage needs, unload after the last stage that
  uses it (~7 GB RAM headroom)
- **Catastrophic abort** for KeyboardInterrupt, disk pressure, DuckDB
  I/O errors â€” preserve `stages_done`, mark run FAILED, re-raise so
  the CLI emits the right exit code

The CLI at `musahit/pipeline.py` is the operator's entry point â€” the
target of the Task Scheduler `MUSAHIT_NIGHTLY` task per ADR-007.

## âťŻ Implementation Approach

### Stage protocol

```python
class Stage(Protocol):
    async def run(self, run_id: str) -> Any: ...
```

Every existing per-stage orchestrator already matches this shape:
`IngestPoller.run(run_id=None)`, `Normalizer.run(run_id)`,
`Clusterer.run(run_id)`, `Classifier.run(run_id)`, `ArcLinker.run(run_id)`,
`Briefer.run(run_id)`, `Synthesizer.run(run_id)`. Return values vary
(stages return dict summaries) but the orchestrator ignores them â€” the
authoritative state lives in DuckDB. Protocol return type is `Any` so
dict returns satisfy the structural contract.

### Orchestrator flow

```
upsert pipeline_runs row (status=RUNNING)
  â†“
read existing stages_done
  â†“
for each stage in STAGE_ORDER:
  if stage in stages_done and not force: skip
  if only_stage set and stage != only_stage: skip
  for model_attr in _MODELS_FOR_STAGE[stage]:
    ollama.load(getattr(settings, model_attr))
  try:
    stage = stage_factory(stage_name)             # lazy construct
    await asyncio.wait_for(
      stage.run(run_id),
      timeout=budget.timeout_seconds,             # 2Ă— soft budget
    )
    append stage to stages_done                   # checkpoint
  except (KeyboardInterrupt, DiskPressureError, duckdb.IOException):
    re-raise as _CatastrophicError                # catastrophic
  except TimeoutError | Exception:
    print traceback to stderr                     # diagnostic
    record failure in failed_stages               # soft failure
    continue                                      # next stage
  for model in models_to_unload_after(stage):
    ollama.unload(...)
  â†“
mark pipeline_runs.status = COMPLETED + completed_at = utcnow()
return PipelineResult
```

### Pre-flight disk check

Before any stage runs, the orchestrator checks `shutil.disk_usage(data_dir)`
against `Settings.min_free_disk_gb` (default 5 GB). Below the floor â†’
`DiskPressureError` raised immediately; the CLI exits with code 1; the
liveness probe (step 18+) surfaces it via Windows toast per ADR-012 Â§
Disk pressure.

### Ollama model lifecycle

Per ADR-001 Â§ Single Ollama instance the laptop holds at most one
heavy model in memory at a time. The orchestrator's lifecycle is:

```
_MODELS_FOR_STAGE = {
    "ingest":   (),
    "normalize":(),
    "cluster":  ("embed_model",),    # bge-m3
    "score":    ("worker_model",),   # qwen2.5
    "arc-link": (),                  # reuses stored embeddings
    "write":    ("writer_model",),   # trendyol-llm
    "tts":      (),                  # piper, not Ollama
}
```

`OllamaModelManager.load(model)` POSTs `{model, prompt: "", stream: false,
keep_alive: "5m"}` to `/api/generate` â€” empty-prompt warm-up that loads
the model weights into memory. `unload(model)` POSTs the same body with
`keep_alive: 0` which tells Ollama to evict the model immediately.

Load/unload failures (Ollama not running, network blip) are logged at
WARN and swallowed â€” the stage that needs the model will fail with a
specific error and the orchestrator's per-stage failure isolation
handles it. The pipeline does not abort if the lifecycle calls misbehave.

After each stage's `run()` completes (success or soft failure), the
orchestrator unloads the model that stage used. The next stage that
needs Ollama loads its own. Net effect: at most one heavy model loaded
at any moment.

### Failure isolation

Three failure tiers:

1. **Soft failure** (any `Exception` other than the catastrophic set):
   - Print traceback to `sys.stderr` (same diagnostic pattern as
     step 14's `Synthesizer` post-smoke-test improvement)
   - Append `StageFailure(name, reason)` to `failed_stages` JSON
   - Continue to the next stage
   - Pipeline still reaches `COMPLETED` status
2. **Stage timeout** (`asyncio.TimeoutError` after `2 Ă— soft_budget`):
   - Same handling as soft failure, but the reason is
     `"TimeoutError: stage exceeded Ns (2Ă— the M-minute ADR-007 budget)"`
3. **Catastrophic** (`KeyboardInterrupt`, `DiskPressureError`,
   `duckdb.IOException`):
   - Wrap in `_CatastrophicError`, unwind the main loop
   - Mark pipeline_runs.status = FAILED
   - Re-raise the original `KeyboardInterrupt` (exit code 2) or
     return a `PipelineResult` with `status=FAILED` for the others
   - `stages_done` is preserved on disk so resume works

### Stage construction is lazy

The factory is called once per stage *at the moment that stage is
about to run*, not at orchestrator construction. This means:

- A missing Piper voice fails the tts stage specifically rather than
  blocking the whole pipeline at startup
- An Ollama unavailable failure fails cluster (the first Ollama
  consumer) rather than blocking startup
- Constructor failure (e.g., `FileNotFoundError` from
  `PiperPythonClient`) is treated as a soft stage failure: recorded
  in `failed_stages`, next stage runs anyway

### CLI surface

```
python -m musahit.pipeline run    --date today
python -m musahit.pipeline run    --date 2026-05-23 --stage cluster
python -m musahit.pipeline run    --date today --force
python -m musahit.pipeline run    --date today --dry-run
python -m musahit.pipeline status --date today
python -m musahit.pipeline resume --date today
```

`run` is the primary entrypoint (Task Scheduler target). `status`
prints the latest `pipeline_runs` row for a date. `resume` is a
convenience alias for `run` without `--force` / `--stage` (same
effective behavior â€” kept for operator muscle memory per ADR-007 Â§
Manual operation).

Exit codes:
- `0` â€” COMPLETED
- `1` â€” FAILED (catastrophic abort, status command found no row, status
  shows non-COMPLETED)
- `2` â€” SIGINT (operator pressed Ctrl-C / process received SIGINT)

`configure_logging()` runs first thing in `main()` so the orchestrator
and stages emit JSON logs to the configured destination.

### Schema migration

Migration `003_add_failed_stages.sql` adds one nullable TEXT column to
`pipeline_runs`. Additive â€” existing rows get NULL. Backward compatible
with the FILE-PROTECTED `IngestPoller` which doesn't reference the
column. The DuckDB `ADD COLUMN IF NOT EXISTS` form is idempotent so
re-running migrations is safe.

The migration runner globs `[0-9][0-9][0-9]_*.sql` and applies them in
lexical order, so adding `003_*.sql` was sufficient â€” no changes to
`migrations.py`. The `test_init_db.py` assertion that pinned
`migrations_applied == 2` updated to `== 3` (5 occurrences).

## âťŻ Mathematical / Statistical Details

### Timing budget arithmetic

Each `StageTimingBudget` has one parameter: `soft_minutes: float`.

```
soft_seconds   = soft_minutes Ă— 60
timeout_seconds = soft_minutes Ă— 60 Ă— 2     # hard timeout = 2 Ă— soft
```

The float type lets tests use fractional minutes (e.g.,
`soft_minutes=0.0005` â†’ 30 ms soft / 60 ms hard) so timeout-path
assertions complete in tens of milliseconds.

ADR-007 Â§ Pipeline timing budget defines six soft deadlines in whole
minutes (ingest 60, normalize 30, cluster 60, score 60, arc-link 30,
write 60). TTS shares the 30-minute slot for "voice synthesis" per the
same ADR. The orchestrator's STAGE_BUDGETS map encodes these directly.

The 06:30 â†’ 06:45 "artifact lock + backup" window in ADR-007 is NOT a
stage in `STAGE_ORDER` â€” it belongs to the post-tts liveness probe
(step 18) and the backup script (later step).

### Resume invariant

`stages_done` is a JSON list of completed-stage names. The orchestrator
appends only on `await asyncio.wait_for(stage.run(...))` returning
without exception; soft failures and timeouts do NOT append (the stage
did not produce its expected output). Catastrophic aborts also do not
append the in-flight stage.

Invariant: after any orchestrator exit (clean or otherwise),
`stages_done` reflects exactly the set of stages whose `run()` returned
without raising. Re-running with the same `run_id` skips this set
and processes only the remainder â€” the ADR-007 Â§ Resumability
contract.

## âťŻ Design Decisions

### Why a `stage_factory` Callable rather than a list of pre-built Stages

Stages need DB connections, Ollama clients, and Piper voices â€”
expensive constructors that may fail. Building all seven at startup
would:

1. Fail the entire pipeline if ANY constructor failed (e.g., Piper
   voice missing â†’ no normalization either)
2. Open three Ollama HTTP clients even when most stages don't use them
3. Make `--stage NAME` mode wasteful (instantiating 6 unused stages)

The lazy `stage_factory(name) -> Stage` pattern avoids all three. The
default production factory builds one stage at a time, the moment
it's needed. Tests inject a `lambda name: precomputed_fakes[name]`
for full control.

### Why an OllamaModelManager class (rather than free functions)

The lifecycle calls share base URL + timeout + retry behaviour. A
class encapsulates them naturally. Tests inject a
`_RecordingOllama` stub that captures `(load, model)` and
`(unload, model)` tuples for assertion. The production class swallows
network failures because the stage that needs the model will surface
the real error with better context.

### Why warm-via-empty-prompt rather than `/api/show`

`/api/generate` with `prompt: ""` is the canonical Ollama warm-up
trick: it forces the runtime to mmap the model into memory and run
the first prompt-template-only forward pass. `/api/show` returns
metadata WITHOUT loading the model. Once the model is warm, the next
stage's real prompt gets first-token latency in tens of milliseconds
instead of ~30 seconds (the cold-load time for a quantized 7B model
on this hardware).

### Why dict-shape `failed_stages` rather than JSON list of strings

Each failure carries two fields the operator needs at 07:00: the
stage name and the reason (exception type + message, or TimeoutError
with the budget). `{"name": "score", "reason": "..."}` keeps the
operator-readable structure intact when the SÄ°STEM LOG renderer
(step 16+) walks it.

### Why catastrophic abort for `duckdb.IOException`

A DuckDB I/O error means the DB file is corrupted, on a failed disk,
or has been locked by another process. Continuing to the next stage
would corrupt state further (or just compound the same error). The
liveness probe at 06:45 will catch the missing COMPLETED status and
alert. Stage-level isolation is the wrong tool â€” we want to stop
trying to write to a broken file.

### Why dry-run uses a separate StageFactory rather than a flag

Tests inject one factory; `--dry-run` swaps to a different factory
internally. This means a real operator running `--dry-run` doesn't
accidentally get partially-real stages. The `_NoOpStage` returned
by `DryRunStageFactory` doesn't touch DuckDB, Ollama, or the
filesystem â€” so dry-run is a true smoke test of stage dispatch and
CLI plumbing without side effects. Pipeline_runs row is NOT created
in dry-run mode (no DB writes at all).

### Alternative considered: separate Stage class per stage

Rejected. The seven stages already exist with their own classes and
test suites â€” `IngestPoller`, `Normalizer`, etc. Wrapping each in
another `_PipelineStage_Ingest` wrapper would duplicate constructor
plumbing and shift complexity sideways. The Protocol-based dispatch
matches the project's existing pattern (Ingester, LlmClient,
EmbeddingClient, PiperClient) and lets each stage retain its own
identity.

### Alternative considered: model lifecycle as a context manager

Rejected. The lifecycle isn't symmetric â€” `bge-m3` loaded by `cluster`
is used by `arc-link` if `arc-link` ever needs it (currently doesn't
per the goal). A flat dict + per-stage load + post-stage unload is
explicit and easy to amend if the model graph changes.

### Alternative considered: separate `failed_stages` table

Rejected for v0.1. The JSON-on-pipeline_runs approach mirrors how
`stages_done` and `counts` already work. A separate audit table can
land later if the failure history needs richer queries â€” for now the
operator reads the SÄ°STEM LOG section of the briefing.

## âťŻ Operator caveats

- **Task Scheduler XML.** ADR-007's `scripts/task_scheduler.xml` lives
  in step 16 or later; the CLI is the target but the actual scheduler
  registration isn't part of this step.
- **Ollama not running.** The orchestrator's load/unload calls swallow
  network failures with a WARN log; the stage that needs the model
  fails with a specific error instead. To diagnose, start Ollama
  manually (`ollama serve`) before running the pipeline.
- **Disk pressure.** Default floor is 5 GB. If `data_dir` is on a
  separate drive with less free space, override `min_free_disk_gb`
  in `config.toml`.
- **Dry-run skips DB writes.** No `pipeline_runs` row is created in
  dry-run mode â€” by design. This means `status` after a dry-run shows
  "no pipeline_runs row" rather than the dry-run's NoOp completion.
- **`failed_stages` is JSON.** Operators reading the column directly in
  `duckdb` REPL should `json_extract` it or parse client-side.
