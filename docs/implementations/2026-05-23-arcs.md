# Implementation: Story arc linking · `musahit/arcs/`

**Date** · 2026-05-23
**Author** · Claude Code (Mert Efe Şensoy directing)
**ADR refs** · ADR-006 · ADR-008 · ADR-012

---

## ❯ Problem / Motivation

Build step 12 of 20. Step 11 produced scored clusters; this stage
collapses sequential clusters about the same underlying event into
**story arcs** per ADR-008. The Machine assigns Numbers; MÜŞAHİT does
the same with story arcs — each arc tracks one ongoing thread (a court
case, a policy shift, a regulatory probe) across days/weeks. The
briefing's "AÇIK GELİŞMELER" section (ADR-009) reads from arcs, not
from clusters.

Three properties carry the design:

1. **Two-filter match.** Centroid cosine ≥ 0.55 *AND* Jaccard ≥ 0.4 on
   stopword-filtered entity sets. Either signal alone misfires often
   enough (centroid drift; entity overlap on common Turkish names)
   that we require both.
2. **Severity-ordered seeding.** Process clusters in `final_defcon`
   ascending order — lower integer = more severe — so an ACUTE event
   seeds the arc and a downstream ROUTINE follow-up matches into it,
   not the reverse.
3. **State transitions** automatic via the cleanup pass that runs at
   the end of every `ArcLinker.run`. OPEN > 7 days → WATCH; WATCH >
   30 days → RESOLVED. Reverse transitions are linker-side (WATCH →
   OPEN on a fresh link) or dashboard-side (RESOLVED → OPEN on
   operator override).

## ❯ Implementation Approach

### The cosine + Jaccard two-filter pattern

```python
def match_arc(centroid, entities, arcs, cosine_threshold=0.55, jaccard_threshold=0.4):
    if not entities:
        return None
    for arc_id, _cosine in find_candidate_arcs(centroid, arcs, cosine_threshold):
        candidate = arcs[arc_id]
        if jaccard(entities, candidate.entity_set) >= jaccard_threshold:
            return arc_id
    return None
```

The order matters: we filter by cosine first (cheap O(1) per arc),
sort high-to-low, then test Jaccard on each candidate top-down. The
*highest cosine that also passes Jaccard* wins. If the top-cosine arc
fails Jaccard, the function falls back to lower-cosine candidates
until it finds one that passes both — this captures the "shared topic,
different framing" case where the embedding may favor a different arc
than the entity set does.

ADR-008 wrote "first match wins · arcs are returned by recency"; we
sort by cosine instead because cosine is the better discriminator in
our test fixtures. The recency ordering can be reimposed by changing
the sort key in `find_candidate_arcs` — single-line edit if first-
month operation surfaces a need.

### Stopwords rationale

Without `STOPWORD_ENTITIES`, every Turkish political cluster mentions
"Türkiye", "Cumhuriyet", "AKP", "CHP", "Hükümet", "Cumhurbaşkanı", etc.
Two unrelated clusters about, say, an İmamoğlu court case and a TCMB
rate decision would each have those generic names in their entity set
and trivially clear the Jaccard threshold against the OTHER arc. Every
arc would merge with every other; the arc model would collapse to one
mega-arc per category.

The curated stopword list (15+ entries) covers the country name,
generic political vocabulary, and the four largest parties. It is
operator-edited; adding a new stopword is a one-line change in
`stopwords.py` when an arc starts mis-merging.

### State-transition cleanup pass timing

`transition_states(conn, now)` runs at the **end** of every
`ArcLinker.run`, after every cluster has been processed. This ordering
matters:

- If we ran the cleanup first, an OPEN arc that today's run would
  match into would be demoted to WATCH before the link, and the
  briefing would lose the "active update" signal.
- Running cleanup last means freshly-linked arcs stay OPEN (the
  linker's WATCH → OPEN promotion is the inverse transition), and
  only arcs that received zero new clusters today face the demotion.

The cleanup uses the same `last_update_at` field the briefing reads,
so the dashboard's OPEN/WATCH/RESOLVED tabs are always consistent
with what the linker decided this run.

### FK workaround usage

The DuckDB FK pattern from `memory/MEMORY.md` § "DuckDB FK Update
Pattern" applies in two places in this stage:

1. **`clusters.arc_id` updates.** `cluster_articles` and
   `cluster_embeddings` reference `clusters.id`. Updating any column
   on `clusters` (including `arc_id`) fires the FK check. Workaround:
   DELETE both children, UPDATE clusters, re-INSERT children. See
   `_update_cluster_arc_id` and `_update_cluster_arc_id_to_value`.
2. **`arcs.*` updates.** `arc_centroids` references `arcs.id`. Same
   pattern: DELETE arc_centroids, UPDATE arcs, re-INSERT centroid.
   See `_update_arc`.

Each statement auto-commits (no explicit `BEGIN`/`COMMIT`). Inside a
transaction the FK check still fires.

A subtle point: clusters that already point at the arc we're updating
ALSO reference `arcs.id`. We temporarily null those `cluster.arc_id`
values during the arc UPDATE and restore them afterward — using the
cluster-level workaround on each affected cluster. The implementation
calls `_update_cluster_arc_id_to_value(cluster_id, None)` then
`_update_cluster_arc_id_to_value(cluster_id, arc_id)` per member.

### `peak_defcon` uses `min`, NOT `max` — severity direction note

ADR-008 § "Arc updates" says:

> `arc.peak_defcon` is updated to `max(arc.peak_defcon, cluster.final_defcon)`

This is the **same prose-bug pattern** as the original ADR-005 formula
that was corrected in the 2026-05-23 amendment. The DEFCON IntEnum
direction (lower integer = more severe) means *peak severity* is the
*smallest integer* ever assigned. To preserve the highest severity
ever seen in the arc, the update operator must be `min`, not `max`.

The implementation uses `min` with an inline comment pointing at the
ADR-005 precedent. The test `TestPeakDefconMinDirection` pins both
cases: a SEVERE cluster joining a MATERIAL-peak arc promotes the peak
to SEVERE (`min(3, 2) = 2`); a ROUTINE cluster joining a SEVERE-peak
arc does NOT regress the peak (`min(2, 4) = 2`).

**Flagged but not amended in this commit.** ADR-008's prose retains
the `max` wording. The follow-up ADR amendment to correct ADR-008's
prose (matching the ADR-005 precedent) is queued; this implementation
doc is the bridge until then. Per the MEMORY.md convention "ADR
semantic intent overrides formula text", the implementation matches
the intent, not the literal formula.

### Severity-ordered seeding

`_select_pending` orders clusters by `final_defcon ASC`. Lower DEFCON
integer = more severe. The test `TestSeverityOrdering` proves the
effect:

- Two clusters with identical centroids and overlapping entities, one
  ACUTE (1) and one ROUTINE (4), share the same arc.
- The ACUTE one seeds; the ROUTINE one matches in.
- The arc's headline comes from the ACUTE cluster (the severe one is
  what the operator should see first).

If processing order were reversed, the ROUTINE cluster would seed an
arc with a generic ROUTINE-grade headline, and the ACUTE cluster
would later match into that arc — losing the headline's signal.

## ❯ Mathematical / Statistical Details

**Cosine + Jaccard combined.** Given a cluster with centroid
`c` and stopword-filtered entity set `E_c`, and an existing arc with
centroid `a` and entity set `E_a`, the cluster matches the arc iff:

```
cosine(c, a) ≥ θ_cos      AND      |E_c ∩ E_a| / |E_c ∪ E_a| ≥ θ_jac
```

with `θ_cos = 0.55` and `θ_jac = 0.4` from ADR-008. The empty-entity
case (`E_c = ∅` after stopword filtering) short-circuits to "no
match" because Jaccard is undefined for `|∪| = 0` and there is no
disambiguating signal to validate the centroid match.

Among arcs that pass both filters, the **highest cosine** wins. This
is an O(N_arcs) scan per cluster; at typical volume (≤30 active arcs
× ≤100 clusters per night = ~3k cosine ops × 1024 dims = ~3M FLOPs)
the cost is below 10ms total. HNSW-based candidate retrieval is
available via the existing VSS index but unnecessary at this scale.

**Centroid recomputation.** After linking, the arc's centroid is the
arithmetic mean of all member cluster centroids. We re-query
`cluster_embeddings` joined on `clusters.arc_id == this_arc_id` so the
result is canonical regardless of which clusters were already linked.
The mean of unit-length vectors is generally NOT unit-length itself;
`cosine_similarity` handles that correctly by computing norms inline.

## ❯ Design Decisions

**Single in-memory cache of active arcs per run.** `_load_arc_cache`
runs once at the start of `run()` and reuses the result for every
cluster. New arcs seeded mid-run are added to the cache so subsequent
clusters can match them. The alternative — re-querying the DB per
cluster — would be O(N_clusters · N_arcs) DB hits when O(N_clusters)
suffices.

**Greedy "first cluster wins" arc updates within a run.** When two
clusters in the same run match the same arc, the first one in the
severity-ordered queue updates the arc; the second one matches into
the already-updated arc. The arc's `last_update_at`, `peak_defcon`,
`entity_set`, and `centroid` all reflect both clusters by the end of
the run. No re-sorting of the cluster queue mid-pass.

**`find_candidate_arcs` returns a list, not a generator.** Sortable,
inspectable, easy to test. The candidate count is small (≤30) so
materialising the list costs nothing.

**Embeddings read straight from the DB.** No Ollama call in this
stage. Cluster centroids land in `cluster_embeddings` from step 10;
arc centroids land in `arc_centroids` from this stage. The
`FakeEmbeddingClient` pattern from step 10 is *not* invoked here
because we never embed anything new — we only compare and average
existing vectors.

**Two FK-workaround entry points.** `_update_cluster_arc_id` (sets a
specific arc_id) and `_update_cluster_arc_id_to_value` (sets any
arc_id including None). The duplication is a slight repetition but
keeps the two callers' intents clear; merging would force a
nullable-arg-shaped function and complicate the read.

**Stopwords list is a frozenset.** Immutable at runtime; the operator
edits the source file and reruns. No "add stopword via dashboard" UI
because the list is small and slow-changing — touching it weekly is
fine.

## ❯ Operator caveats

- **First-month tuning.** Thresholds `0.55` cosine / `0.4` Jaccard are
  starting guesses. Watch the dashboard for:
  - **Under-linking** (many small new arcs about the same event):
    lower one or both thresholds.
  - **Over-linking** (one arc that absorbs everything): raise the
    Jaccard threshold first; raise cosine if it doesn't help.
- **MERGE / SPLIT are operator actions.** The linker is conservative;
  ADR-008 explicitly expects manual MERGE actions to be common in the
  first month, especially during the bootstrap period when there's
  no arc history to link new clusters to. The dashboard surfaces
  arcs that look mergeable.
- **Stopword maintenance.** Adding a stopword is a one-line edit in
  `stopwords.py`. Watch for arcs that look like they shouldn't have
  linked — if the only shared entities are generic political nouns,
  promote those nouns to stopwords.
- **`peak_defcon` direction.** Lower DEFCON integer = more severe.
  The arc's peak is the **smallest integer** ever seen. The
  implementation uses `min` despite ADR-008's `max` prose — see the
  "Severity direction" section above. Don't be confused if you read
  ADR-008 first.
- **HNSW unused (today).** Cosine over 30 active arcs is trivial.
  When the active-arc count exceeds ~500, switching to the HNSW index
  in `arc_centroids` is the right move. The index already exists
  (created by `init_db` when VSS is available).
- **The cleanup pass is non-reversible for RESOLVED.** Once an arc
  transitions to RESOLVED via the 30-day silence rule, the linker
  does not re-open it on a future link — the operator must reopen
  via the dashboard. Per ADR-008.
